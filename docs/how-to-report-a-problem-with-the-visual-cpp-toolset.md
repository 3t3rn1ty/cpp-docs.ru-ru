# <a name="how-to-report-a-problem-with-the-visual-c-toolset"></a>Как сообщить о проблеме с набором инструментов Visual C++

Если у вас возникают проблемы с компилятором, компоновщиком или другими инструментами Visual C++, мы бы хотели узнать об этом.

Лучший способ сообщить нам о проблеме — отправить нам отчет с ее описанием, сведениями о том, как выполняется сборка программы, и кодом, с помощью которого мы сможем воспроизвести проблему на собственных компьютерах. Эта информация позволит нам быстро выяснить, действительно ли проблема существует (то есть не является характерной лишь для вашей среды), определить, затрагивает ли она другие версии компилятора, и установить ее причину.

В этом документе вы узнаете:

- [Как подготовить отчет](#prepare) и как сделать его качественным.

- [Как создать данные для воспроизведения](#generate) и какие типы таких данных возможны.

- [Как отправить отчет](#send) и в чем отличия разных способов отправки.

Ваши отчеты важны для нас и других разработчиков. Благодарим за помощь в улучшении Visual C++!

## <a name="prepare"></a> Подготовка отчета

Создание качественного отчета имеет важное значение, так как не имея полной информации, нам будет очень трудно воспроизвести возникшую у вас проблему на собственных компьютерах. Чем более исчерпывающим будет ваш отчет, тем эффективнее мы сможем воссоздать и диагностировать проблему.

Отчет должен содержать по меньшей мере следующие данные:

- полная информация о версии используемого набора инструментов;

- полная команда cl.exe, используемая для сборки кода;

- подробное описание возникшей проблемы;

- данные для воспроизведения — исходный код, демонстрирующий проблему.

Далее подробнее описывается то, какая информация нам нужна и где ее можно найти.

### <a name="the-toolset-version"></a>Версия набора инструментов

Нам нужна полная информация о версии используемого набора инструментов, чтобы мы могли протестировать ваш код с помощью той же версии на наших компьютерах. Если мы сможем воспроизвести проблему, то затем попробуем выяснить, в каких еще версиях набора инструментов она проявляется.

#### <a name="to-report-the-full-version-of-the-compiler-youre-using"></a>Предоставление информации о полной версии используемого компилятора

1. Нажмите клавишу Windows и начните вводить команду `Developer Command Prompt`.

1. Выберите версию **Командной строки разработчика**, соответствующую используемой версии Visual Studio, когда она появится в списке соответствий.

1. В консоли **Командная строка разработчика** введите команду `cl /Bv /CLR`.

Результат должен выглядеть следующим образом:


Скопируйте все выходные данные и вставьте их в отчет.

### <a name="the-command-line"></a>Командная строка

Нам нужна полная команда (cl.exe и все аргументы), которую вы используете для сборки кода, чтобы мы могли выполнить сборку аналогичным образом на своих компьютерах. Это важно, так как проблема может возникать только при сборке с определенным аргументом или сочетанием аргументов.

Проще всего найти эту информацию в журнале сборки сразу после возникновения проблемы. Таким образом мы получим команду именно с теми аргументами, которые вы использовали и которые могут быть причиной проблемы.

#### <a name="to-report-the-contents-of-the-command-line"></a>Предоставление содержимого командной строки

1. Найдите файл **CL.command.1.tlog** и откройте его. По умолчанию этот файл находится в папке \\...\*версия* Visual Studio\Projects\\*Имя_решения*\\*Имя_проекта*\Config\\*Имя_проекта*.tlog\CL.command.1.tlog.

   В этом файле в отдельных строках указываются имена файлов с исходным кодом и аргументы командной строки, которые использовались для их компиляции.

1. Найдите строку с именем файла, в котором возникла проблема. В следующей строке содержится соответствующая команда cl.exe и ее аргументы.

Скопируйте командную строку полностью и вставьте ее в отчет.

### <a name="a-description-of-the-problem"></a>Описание проблемы

Нам требуется подробное описание возникшей проблемы, чтобы мы могли проверить, проявляется ли она на наших компьютерах. Иногда нам также полезно знать, какую задачу вы выполняли и какого результата ожидали.

Предоставьте нам сообщения об ошибках, которые выводились набором инструментов, краткое описание выполняемой задачи, чтобы мы могли понять ваш исходный код, и другие сведения, которые могут помочь нам в диагностике проблемы, например найденные вами обходные решения. Старайтесь не указывать повторно сведения, которые уже имеются в других частях отчета.

### <a name="the-repro"></a>Данные для воспроизведения

Нам требуются *данные для воспроизведения* — автономный образец исходного кода, демонстрирующий возникшую проблему и позволяющий нам воспроизвести ее на своих компьютерах. Данные для воспроизведения, которые следует включить в отчет, зависят от типа возникшей проблемы. Не имея этих данных, мы не сможем провести анализ.

Данные для воспроизведения небольшого объема можно включить непосредственно в текст отчета, но более крупные образцы исходного кода следует отправлять вложением. Если данные для воспроизведения нельзя поместить в один файл с исходным кодом, их следует упаковать путем сжатия каталога, содержащего все файлы, в ZIP-файл или файл аналогичного формата, а затем вложить в отчет. Дополнительные сведения о ситуации, в которой возникает ошибка, следует включать в текст отчета, но не в исходный код.

В идеале нам следует предоставить *минимальные необходимые данные для воспроизведения*. Это один автономный файл с исходным кодом (без ссылок на пользовательские заголовки), который содержит ровно столько кода, сколько нужно для воспроизведения проблемы. Если вы можете предоставить данные для воспроизведения в такой форме, просто вложите файл исходного кода в отчет — это все, что нам нужно.

Если необходимые данные для воспроизведения не удается свести к одному файлу без зависимостей, обратитесь к следующим разделам, чтобы определить, какие данные следует включить в отчет.

#### <a name="frontend-parser-crash"></a>Сбой интерфейсной части (средства синтаксического анализа)

Сбои интерфейсной части происходят на этапе выполнения синтаксического анализа компилятором. Как правило, компилятор выдает [неустранимую ошибку C1001](error-messages/compiler-errors-1/fatal-error-c1001.md) с указанием файла с исходным кодом и номера строки, в которой произошла ошибка. Часто указывается файл msc1.cpp, однако эту информацию можно игнорировать.

В случае такого сбоя предоставьте [предварительно обработанные данные для воспроизведения](#preprocessed-repros).

Вот пример выходных данных компилятора для сбоя такого типа:

```Output

Internal Compiler Error in d:\o\dev\otools\bin\x64\cl.exe.  You will be prompted
```

#### <a name="backend_crash"></a> Внутренний сбой (при создании кода)

Внутренние сбои происходят на этапе создания кода компилятором. Как правило, компилятор выдает [неустранимую ошибку C1001](error-messages/compiler-errors-1/fatal-error-c1001.md), возможно, без указания файла с исходным кодом и номера строки, в которой произошла ошибка. Часто указывается файл compiler\utc\src\p2\main.c, однако эту информацию можно игнорировать.

В случае такого сбоя предоставьте [данные компоновки для воспроизведения](#link-repros), если вы используете создание кода во время компоновки (LTCG), или [предварительно обработанные данные для воспроизведения](#preprocessed-repros) в противном случае. LTGC включается с помощью аргумента командной строки `/GL` для команды cl.exe.

```Output

        An internal error has occurred in the compiler.
(compiler file 'f:\dd\vctools\compiler\utc\src\p2\main.c', line 230)
To work around this problem, try simplifying or changing the program near the
        locations listed above.
Please choose the Technical Support command on the Visual C++
Help menu, or open the Technical Support help file for more information

        'C:\Program Files (x86)\Microsoft Visual Studio 14.0\VC\BIN\cl.exe'
```

Если в строке, которая начинается со слов **ВНУТРЕННЯЯ ОШИБКА КОМПИЛЯТОРА**, упоминается программа link.exe, а не cl.exe, функция LTCG была включена и вам следует предоставить [данные компоновки для воспроизведения](#link-repros). Если из сообщения об ошибке компилятора неясно, была ли функция LTCG включена, может потребоваться изучить аргументы командной строки, которые вы скопировали из журнала сборки ранее, и проверить наличие аргумента `/GL`.

#### <a name="bad-code-generation"></a>Создание неверного кода

Хотя такое происходит нечасто, компилятор может по ошибке создавать неверный код, который не определяется во время компиляции, но приводит к сбою приложения во время выполнения. Если вы считаете, что возникающая проблема приводит к созданию неверного кода, следуйте указаниям, приведенным в разделе [Внутренний сбой (при создании кода)](#backend_crash).

В случае такого сбоя предоставьте [данные компоновки для воспроизведения](#link-repros), если вы используете создание кода во время компоновки (LTCG), или [предварительно обработанные данные для воспроизведения](#preprocessed-repros) в противном случае. LTGC включается с помощью аргумента командной строки `/GL` для команды cl.exe.

## <a name="generate"></a> Создание данных для воспроизведения

Данные для воспроизведения — это полный, автономный образец кода, демонстрирующий проблему, о которой вы сообщаете. Данные для воспроизведения — это **не** фрагмент кода. Они должны представлять собой полный образец, который можно собрать и выполнить (или можно было бы, если бы не ошибки, возникающие из-за проблемы). Код должен содержать все необходимые директивы #include, даже для стандартных заголовков.

Кроме того, желательно, чтобы данные для воспроизведения отвечали указанным ниже требованиям.

- **Минимальный размер**. Данные для воспроизведения должны иметь как можно меньший размер, но при этом точно демонстрировать проблему. Они не обязательно должны быть сложными или приближенными к реальности — старайтесь делать их проще и включать только то, что относится к сути. Включать для сравнения примеры правильно работающего кода необязательно, но можно, если это повышает наглядность. Обязателен только код, вызывающий проблему.

- **Автономность**. В данных для воспроизведения следует избегать лишних зависимостей. Если проблему можно воспроизвести без сторонних библиотек, то не используйте их. Если проблему можно воспроизвести вообще без кода библиотек (библиотеки `std::out` и `printf()` допустимы), то не включайте его. Сокращая объем кода, который может вносить вклад в возникновение проблемы, вы оказываете нам большую услугу.

- **Использование последней версии компилятора**. По возможности данные для воспроизведения должны использовать последнюю версию набора инструментов. Многие проблемы, которые могут возникать в предыдущих версиях набора инструментов, исправлены в его более новых версиях.

- **Тестирование с другими компиляторами**, если это необходимо. Данные для воспроизведения, включающие переносимый код C++, следует протестировать с другими компиляторами, если это возможно.

   Это позволяет определить правильность кода, если, например, результаты для MSVC расходятся с результатами для Clang и GCC, или его неправильность, если результаты для MSVC, Clang и GCC сходятся и подтверждают наличие ошибки.

Ниже приведены инструкции по созданию различных типов данных для воспроизведения в зависимости от характера проблемы.

### <a name="preprocessed-repros"></a>Предварительно обработанные данные для воспроизведения

Предварительно обработанные данные для воспроизведения — это отдельный файл с исходным кодом, который демонстрирует проблему и был создан на основе выходных данных препроцессора C путем обработки исходного файла. В рамках этого процесса происходит внедрение включенных заголовков для устранения зависимостей от дополнительных файлов с исходным кодом и файлов заголовков, а также разрешение макросов, директив #ifdef и других команд препроцессора, которые могут зависеть от вашей локальной среды.

> [!NOTE]
> Обратите внимание на то, что предварительно обработанные данные для воспроизведения могут быть не так полезны, если проблема является результатом ошибок в нашей стандартной реализации библиотек, так как мы часто используем последнюю реализацию, чтобы проверить, была ли проблема уже устранена. В этом случае не выполняйте предварительную обработку данных. Если проблему нельзя продемонстрировать с помощью одного файла с исходным кодом, упакуйте код в ZIP-файл или файл аналогичного формата или предоставьте образец в виде проекта интегрированной среды разработки (см. раздел [Другие типы данных для воспроизведения](#other-repros)).

#### <a name="to-preprocess-a-source-code-file"></a>Предварительная обработка файла с исходным кодом

1. Нажмите клавишу Windows и начните вводить команду `Developer Command Prompt`.

1. Выберите версию **Командной строки разработчика**, соответствующую используемой версии Visual Studio, когда она появится в списке соответствий.

1. В окне консоли **Командной строки разработчика** введите команду `cl /P argumentsfilename.cpp`.

После того как вы получите предварительно обработанный файл (имя_файла.i), рекомендуется проверить, воспроизводится ли проблема по-прежнему с его помощью. С помощью аргумента командной строки `/TP` можно указать, что программа cl.exe должна пропустить этап препроцессора и попытаться скомпилировать код обычным образом.

#### <a name="to-confirm-that-the-error-still-repros-with-the-preprocessed-file"></a>Проверка воспроизведения ошибки с помощью предварительно обработанного файла

1. Нажмите клавишу Windows и начните вводить команду `Developer Command Prompt`.

1. Выберите версию **Командной строки разработчика**, соответствующую используемой версии Visual Studio, когда она появится в списке соответствий.

1. В окне консоли **Командной строки разработчика** введите команду `cl arguments /TP filename.i`.

1. Убедитесь в том, что проблема воспроизводится.

Наконец, вложите данные для воспроизведения в отчет.

### <a name="link-repros"></a>Данные компоновки для воспроизведения

Данные компоновки для воспроизведения — это отдельный каталог с артефактами сборки, которые в совокупности демонстрируют проблему, возникающую во время компоновки. Это может быть внутренний сбой при использовании функции создания кода во время компоновки (LTCG) или сбой компоновщика. Включать следует только те артефакты сборки, которые требуются в качестве входных данных компоновщика для воспроизведения проблемы. Данные компоновки для воспроизведения можно легко создать с помощью вспомогательных средств, предоставляемых компоновщиком.

#### <a name="to-generate-a-link-repro"></a>Создание данных компоновки для воспроизведения

1. Откройте командную строку и введите команду `mkdir directory`, чтобы создать каталог для данных компоновки.

1. В качестве значения переменной среды link_repro укажите созданный каталог, введя команду `set link_repro=directory`.

1. Если вы хотите выполнить сборку из среды Visual Studio, запустите ее из командной строки с помощью команды `devenv`. Таким образом, значение переменной среды link_repro будет доступно Visual Studio.

1. Выполните сборку приложения и проверьте, возникает ли проблема.

1. Если в шаге 3 вы открыли среду Visual Studio, закройте ее.

1. Очистите переменную среды link_repro, выполнив команду `set link_repro=`.

Наконец, упакуйте данные для воспроизведения, сжав весь каталог в ZIP-файл или файл аналогичного формата, и вложите их в отчет.

### <a name="other-repros"></a>Другие типы данных для воспроизведения

Если проблему не удается свести к одному файлу с исходным кодом или представить в виде предварительно обработанных данных, а данные компоновки не требуются, мы можем изучить проект интегрированной среды разработки. Код проекта по-прежнему должен иметь минимальный размер, и к нему применимы все рекомендации, приведенные в этом документе.

Создайте данные для воспроизведения в виде проекта интегрированной среды разработки минимального размера, упакуйте его, сжав всю структуру каталогов в ZIP-файл или файл аналогичного формата, и вложите в отчет.

## <a name="send"></a> Способы отправки отчета

Вы можете доставить нам отчет несколькими способами. Можно воспользоваться встроенным в Visual Studio [инструментом "Сообщить о проблеме"](/visualstudio/ide/how-to-report-a-problem-with-visual-studio-2017) или [страницами сообщества разработчиков Visual Studio](https://developercommunity.visualstudio.com/). Также можно отправить сообщение электронной почты с отчетом, но предпочтительными являются первые два метода. Вариант зависит от желательного способа взаимодействия со специалистами, которые будут исследовать отчет, и необходимости отслеживать ход работы над отчетом и предоставления его сообществу.

> [!NOTE]
> Независимо от способа отправки отчета корпорация Майкрософт сохраняет вашу конфиденциальность. Сведения о том, как мы обращаемся с поступающей от вас информацией, см. в [Заявлении о конфиденциальности для семейства продуктов Microsoft Visual Studio](https://www.visualstudio.com/dn948229).

### <a name="use-the-report-a-problem-tool"></a>Использование инструмента "Сообщить о проблеме"

Инструмент **Сообщить о проблеме** в Visual Studio позволяет пользователям Visual Studio сообщать о различных проблемах всего несколькими щелчками мыши. Он предоставляет простую форму, в которой можно указать подробные сведения о возникшей проблеме, а затем отправить отчет, не выходя из интегрированной среды разработки.

С помощью инструмента **Сообщить о проблеме** вы можете быстро сообщить о проблеме из интегрированной среды разработки. Этот инструмент доступен в строке заголовка. Чтобы его открыть, щелкните значок **Отправить отзыв** рядом с полем поиска **Быстрый запуск**. Либо найдите его в строке меню, последовательно выбрав **Справка** > **Отправить отзыв** > **Сообщить о проблеме**.

Решив сообщить о проблеме, сначала попытайтесь найти подобный вопрос в сообществе разработчиков. Если проблема уже была зарегистрирована ранее, проголосуйте за тему и добавьте комментарии с дополнительными аспектами. Если подобная проблема отсутствует, нажмите кнопку **Сообщить о новой проблеме**, расположенную в нижней части диалогового окна отзывов Visual Studio, и выполните действия об указании сведений об ошибке.

### <a name="use-the-visual-studio-developer-community-pages"></a>Использование страниц сообщества разработчиков Visual Studio

Страницы сообщества разработчиков Visual Studio являются еще одним удобным способом указания сведений о проблемах и поиска решений по работе с Visual Studio, компилятором C++, средствами и библиотеками. На странице [Вопросы по Visual Studio](https://developercommunity.visualstudio.com/spaces/8/index.html) размещается информация о проблемах с интегрированной средой разработки или установкой. Сведения о проблемах с компилятором C++, компоновщиком и другими средствами и библиотеками публикуются на странице [Вопросы по C++](https://developercommunity.visualstudio.com/spaces/62/index.html).

На баннере сообщества разработчиков, расположенном в верхней части каждой страницы, находится поле поиска, которое можно использовать для поиска сообщений или разделов, содержащих данные о проблемах, подобных вашим. Может оказаться, что решение или другие полезные сведения по вашему вопросу уже доступны в существующем разделе. Если кто-то уже сообщил о подобной проблеме ранее, проголосуйте "за" и оставьте комментарий по этой теме, а не создавайте отчет о проблеме.

Если сведения по вашей проблеме отсутствуют, нажмите кнопку **Сообщить о проблеме** рядом с полем поиска на странице сообщества разработчиков. Вам может быть предложено войти в учетную запись Visual Studio и предоставить доступ приложению сообщества разработчиков к вашему профилю. Выполнив вход, вы сразу попадете на страницу, с которой можно сообщить о проблеме. Вы можете включить код воспроизведения, командную строку, снимки экранов, ссылки на соответствующие обсуждения и любую другую информацию, которую вы считаете актуальной и полезной.
